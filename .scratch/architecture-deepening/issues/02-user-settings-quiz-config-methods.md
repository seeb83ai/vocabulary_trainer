# Add `QuizConfig()` / `NewWordConfig()` methods to `UserSettings`

Status: ready-for-agent
Category: enhancement

## What to build

Add two projection methods to `models.UserSettings` that return the sm2 config structs the quiz handler needs, eliminating the manual field-by-field reconstruction in `QuizHandler.Next()`.

Currently the handler copies 8 fields from `UserSettings` into `sm2.ProgressiveModeConfig` and `sm2.NewWordModeConfig` by hand. If a new mode weight field is added to `UserSettings`, every reconstruction site must be updated. The struct mixes quiz weights, API keys, and language settings — a handler that only needs quiz mode config still reads the full 15-field struct.

## Acceptance criteria

- [ ] `(s *UserSettings) QuizConfig() sm2.ProgressiveModeConfig` method added to `models/models.go` (or `sm2` package — wherever avoids import cycles)
- [ ] `(s *UserSettings) NewWordConfig() sm2.NewWordModeConfig` method added alongside it
- [ ] When `UserSettings` is nil (unauthenticated / first-run), callers use the existing `sm2.DefaultProgressiveModeConfig()` and `sm2.DefaultNewWordModeConfig()` — document this at the call site
- [ ] `QuizHandler.Next()` replaces the manual struct construction with calls to these methods
- [ ] Unit tests assert the projection is correct for a fully-populated `UserSettings` and for the nil/default fallback
- [ ] All existing Go tests pass (`go test ./... -count=1`)

None — can start immediately.

**Category:** enhancement
**Summary:** Add `QuizConfig()` and `NewWordConfig()` projection methods to `UserSettings` so the quiz handler stops reconstructing sm2 config structs by hand.

**Current behavior:**
`QuizHandler.Next()` manually maps 8 fields from `UserSettings` into two sm2 config structs (`ProgressiveModeConfig` and `NewWordModeConfig`). This reconstruction is boilerplate: if a new mode weight field is added to `UserSettings`, every site that builds these structs must be updated. `UserSettings` gains two methods — `QuizConfig() sm2.ProgressiveModeConfig` and `NewWordConfig() sm2.NewWordModeConfig` — that perform the projection internally. `QuizHandler.Next()` calls these methods instead of mapping fields manually. When settings are nil (no user / first run), the handler falls back to the existing `sm2.DefaultProgressiveModeConfig()` and `sm2.DefaultNewWordModeConfig()` — the nil-check stays in the handler, but the struct construction moves to the methods.

- `models.UserSettings` — add `QuizConfig()` and `NewWordConfig()` methods; place them wherever avoids import cycles (if `models` imports `sm2` creates a cycle, consider placing the methods in the `sm2` package as standalone functions accepting `*UserSettings`, or introduce a small config struct in `models`)
- `sm2.ProgressiveModeConfig` and `sm2.NewWordModeConfig` — the return types; no changes to these structs
- `QuizHandler.Next()` — replace the manual field-copy block with calls to the new methods
