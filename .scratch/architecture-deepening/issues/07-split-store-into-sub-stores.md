# Split `Store` into focused sub-stores

Status: ready-for-human
Category: enhancement
Type: HITL — architectural sign-off required before implementation

## What to build

The `Store` struct has 117+ methods across 8 files. Every handler depends on the entire `Store`, even when it only touches one domain. There is no seam between the vocabulary entry domain, the quiz scheduling domain, the mnemonic library domain, and the pinyin training domain. Handler tests must construct a full `Store` even for tests that only touch one domain.

Introduce focused sub-stores so handlers declare dependency on the smallest surface they actually use:

- `WordStore` — vocabulary entry CRUD, translation links, tag queries
- `QuizStore` — SM-2 progress, `GetNextCard`, confusion pairs, daily stats
- `MnemonicStore` — HMM actor/location/room/prop library, scene storage
- `PinyinStore` — pinyin sound library and progress
- `ComponentStore` — hanzi component progress and initialisation
- `UserStore` — user CRUD, authentication, settings

`Store` continues to embed all sub-stores (zero breaking change to construction). Handler structs replace `*db.Store` fields with the narrowest sub-store interface they need.

**HITL decision needed:** confirm the sub-store grouping above before implementation begins. In particular: should `QuizStore` own confusion pairs and daily stats, or should those be on `WordStore`?

## Acceptance criteria

- [ ] Architectural grouping agreed with a human before any code is written
- [ ] Sub-store types defined; `Store` embeds all of them (no change to `db.Open()` callers)
- [ ] Handler structs updated to hold the narrowest sub-store they need (e.g. `QuizHandler` holds `QuizStore`, not `*db.Store`)
- [ ] No SQL moves between files — this is a type-level reorganisation only
