# Unify SM-2 answer processing behind `ProcessAnswer`

Status: ready-for-agent
Category: enhancement

## What to build

Introduce a single `sm2.ProcessAnswer(progress SM2Progress, correct bool) SM2Progress` function that encapsulates the full answer-processing pipeline: the learning-phase branch (`UpdateLearning` vs `Update`), conditional `TotalAttempts` increment, and the `CalcStreakBonus` call. Replace the manual three-step orchestration in `QuizHandler.Answer()` with a single call to this function.

The handler currently leaks SM-2 implementation details: it must know that `UpdateLearning` returns a graduation bool, that `TotalAttempts` must not be incremented when a learning-phase card graduates, and that `CalcStreakBonus` must always run last. All of that belongs inside `sm2.go`.

## Acceptance criteria

- [ ] `sm2.ProcessAnswer(progress SM2Progress, correct bool) SM2Progress` exists in `service/sm2/sm2.go`
- [ ] It handles both the learning-phase and the regular SM-2 path internally
- [ ] `QuizHandler.Answer()` in `service/handlers/quiz.go` calls `sm2.ProcessAnswer` instead of the three-step sequence
- [ ] Unit tests in `service/sm2/` cover: correct answer in learning phase (graduation), wrong answer in learning phase (streak reset), correct answer in regular phase, wrong answer in regular phase
- [ ] All existing Go tests pass (`go test ./... -count=1`)

## Agent Brief

**Category:** enhancement
**Summary:** Introduce `sm2.ProcessAnswer` to hide the learning-phase branching from the quiz answer handler.

**Current behavior:**
`QuizHandler.Answer()` manually orchestrates three SM-2 functions in sequence. It checks `progress.LearningNewWord` to decide whether to call `sm2.UpdateLearning` or `sm2.Update`, conditionally increments `TotalAttempts` (only when `UpdateLearning` did not graduate the card), and always calls `sm2.CalcStreakBonus` last. This means the handler must understand SM-2 internals: which function returns a graduation bool, when to skip the attempts counter, and the invariant that streak bonus must run after everything else.

**Desired behavior:**
A single `sm2.ProcessAnswer(progress SM2Progress, correct bool) SM2Progress` function in the `sm2` package handles all of the above internally. The quiz answer handler calls this one function and receives the fully updated progress struct — no knowledge of the learning phase, graduation, or streak bonus ordering required.

**Key interfaces:**
- `sm2.ProcessAnswer(progress SM2Progress, correct bool) SM2Progress` — new function to add to the `sm2` package
- `sm2.Update`, `sm2.UpdateLearning`, `sm2.CalcStreakBonus` — existing functions that `ProcessAnswer` delegates to internally; they may remain exported for other callers but are no longer called directly by the handler
- `QuizHandler.Answer()` — replace the three-step sequence with a single `sm2.ProcessAnswer` call

**Acceptance criteria:**
- [ ] `sm2.ProcessAnswer(progress SM2Progress, correct bool) SM2Progress` exists in the `sm2` package
- [ ] It handles the learning-phase path (delegates to `UpdateLearning`, skips `TotalAttempts` increment on graduation)
- [ ] It handles the regular SM-2 path (delegates to `Update`, always increments `TotalAttempts`)
- [ ] It always applies `CalcStreakBonus` to the result before returning
- [ ] `QuizHandler.Answer()` is reduced to a single `sm2.ProcessAnswer` call replacing the three-step sequence
- [ ] Unit tests in the `sm2` package cover all four cases: correct in learning phase (graduation), wrong in learning phase (reset), correct in regular phase, wrong in regular phase
- [ ] All existing Go tests pass (`go test ./... -count=1`)

**Out of scope:**
- Changing SM-2 algorithm parameters (`QualityCorrect`, `QualityWrong`, EF formula) — these are frozen per ADR-0002
- Changing how the handler persists the returned progress to the store
- Pinyin or component quiz paths — only the main vocabulary entry quiz handler
