# Consolidate tier classification into a single Go source of truth

Status: ready-for-agent
Category: enhancement

## What to build

The accuracy bucket rules (new, struggling, learning, practicing, mastered) are independently implemented three times: as SQL WHERE fragments in `tierFilter()` in `service/db/words.go`, as a Go switch in `wordTier()` in `service/handlers/quiz.go`, and as a JS copy in `service/frontend/app.js`. A comment in quiz.go admits the synchronisation requirement. One off-by-one in any tier boundary silently breaks quiz filtering, card display, and stats bucketing in different ways.

Introduce a `sm2.Tier` type with named constants and a `sm2.ClassifyTier(progress SM2Progress) Tier` function. Replace `wordTier()` in the handler with a call to `ClassifyTier`. Replace the inline SQL fragments in `tierFilter()` with WHERE clauses generated from the same constants. The JS copy remains (cross-language sync is unavoidable) but a Go test asserts the boundary values used in `ClassifyTier` match a table of expected JS outputs.

## Acceptance criteria

- [ ] `sm2.Tier` type and constants (`TierNew`, `TierStruggling`, `TierLearning`, `TierPracticing`, `TierMastered`) defined in `service/sm2/sm2.go` (or a new `tier.go` in that package)
- [ ] `sm2.ClassifyTier(progress SM2Progress) Tier` implemented with the same boundary logic currently in `wordTier()`
- [ ] `tierFilter()` in `service/db/words.go` generates its SQL using these constants (or a helper that maps `Tier` → SQL fragment), so boundary values are not duplicated
- [ ] `wordTier()` in `service/handlers/quiz.go` is replaced by `sm2.ClassifyTier()`; `Tier.String()` (or equivalent) produces the display label
- [ ] Unit tests in `service/sm2/` assert `ClassifyTier` output for each tier boundary (edge cases: 0 attempts, exactly 3 attempts, exactly 10 attempts, each accuracy threshold)
- [ ] All existing Go tests pass (`go test ./... -count=1`)

## Agent Brief

**Category:** enhancement
**Summary:** Introduce `sm2.Tier` and `sm2.ClassifyTier` as the single Go source of truth for accuracy bucket classification, replacing the duplicated `wordTier()` handler function and the inline SQL fragment in `tierFilter()`.

**Current behavior:**
The accuracy bucket rules are independently implemented in three places: a `wordTier()` switch in the quiz handler (returns display strings), inline SQL CASE expressions in `tierFilter()` (generates WHERE clauses for DB queries), and a JavaScript copy in the frontend. A comment in the handler explicitly warns that all three must stay in sync. A tier boundary change requires editing three files. Any drift between implementations silently breaks quiz filtering, card display, or stats in different ways.

**Desired behavior:**
A `sm2.Tier` type with named constants (`TierNew`, `TierStruggling`, `TierLearning`, `TierPracticing`, `TierMastered`) and a `sm2.ClassifyTier(progress SM2Progress) sm2.Tier` function become the authoritative Go implementation. The handler's `wordTier()` function is deleted and replaced by a call to `ClassifyTier` (with `Tier.String()` producing the display label). The SQL tier filter generates its WHERE clause from the same `Tier` constants — the numeric boundaries (e.g. `0.50`, `0.70`, `0.85`, attempt thresholds `3` and `10`) appear exactly once in Go code. The JS copy remains (cross-language sync is unavoidable), but a table-driven Go test documents the boundary values so any future drift is caught in CI.

**Key interfaces:**
- `sm2.Tier` — new type (consider `type Tier int`) with exported constants for each bucket
- `sm2.ClassifyTier(progress SM2Progress) Tier` — classifies a progress record; encapsulates all boundary logic
- `Tier.String() string` — returns the display label matching what `wordTier()` currently returns (e.g. `"New"`, `"Struggling"`, `"Mastered"`)
- `tierFilter(bucket string) string` in the `db` package — replace inline SQL CASE conditions with values derived from `Tier` constants; the function signature need not change
- `wordTier(p SM2Progress) string` in the handlers package — delete; callers use `sm2.ClassifyTier(p).String()`

**Acceptance criteria:**
- [ ] `sm2.Tier` type and at minimum five constants (`TierNew`, `TierStruggling`, `TierLearning`, `TierPracticing`, `TierMastered`) exist in the `sm2` package
- [ ] `sm2.ClassifyTier(progress SM2Progress) Tier` implements the full boundary logic (handles 0 attempts, learning phase flag, accuracy thresholds at 0.50 / 0.70 / 0.85, attempt thresholds at 3 and 10)
- [ ] `Tier.String()` returns labels matching the current display strings (`"New"`, `"Struggling"`, `"Learning"`, `"Practicing"`, `"Mastered"`)
- [ ] The handler's `wordTier()` function is deleted; all call sites use `sm2.ClassifyTier(...).String()`
- [ ] `tierFilter()` in the `db` package uses `Tier` constants (or a helper derived from them) so the accuracy thresholds are not re-stated in SQL strings
- [ ] Table-driven unit tests in the `sm2` package assert `ClassifyTier` output at every boundary (0 attempts, 2 attempts, 3 attempts, 9 attempts, 10 attempts; accuracies just below and just above 0.50, 0.70, 0.85; `LearningNewWord=true`)
- [ ] All existing Go tests pass (`go test ./... -count=1`)

**Out of scope:**
- Changing or adding tiers — the existing five buckets are preserved exactly
- Modifying the JS frontend — the JS copy stays as-is; only the Go side is unified
- Changing how tier labels are rendered in the UI
- SM-2 parameter values (frozen per ADR-0002)
