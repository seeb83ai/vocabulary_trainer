# Introduce a card scheduler seam in `QuizHandler.Next()`

Status: ready-for-agent
Category: enhancement

## What to build

`QuizHandler.Next()` is ~250 lines that fetch vocabulary entry cards, mnemonic cards, and component cards from three independent store queries, then hand-code priority logic to pick which one to serve. The business rule ("serve the most-overdue card") is implicit in a chain of nested if-statements. Adding a fourth card source requires another query and another branch.

Extract a `SelectNextCard` function that takes a slice of candidate cards (each with a `DueDate` and a typed payload) and returns the one with the earliest due date. Each store query feeds one candidate into the slice. The handler calls `SelectNextCard`, then serialises the chosen card.

The priority logic has a name, a location, and its own tests ‚Äî independent of store setup.

## Acceptance criteria

- [ ] A `SelectNextCard(candidates []CardCandidate) (CardCandidate, bool)` function (or equivalent) exists in a handler-local or `sm2` package
- [ ] `CardCandidate` carries a `DueDate time.Time` and a typed union (vocabulary entry / mnemonic / component) sufficient for the handler to serialise the response
- [ ] `QuizHandler.Next()` builds candidates from the three existing store queries and delegates selection to `SelectNextCard`
- [ ] Unit tests for `SelectNextCard` cover: single source, tie-break (equal due dates), all sources nil/empty
- [ ] The handler's observable behaviour (which card type is served in each scenario) is covered by at least one integration test per card type
- [ ] All existing Go tests pass (`go test ./... -count=1`)
