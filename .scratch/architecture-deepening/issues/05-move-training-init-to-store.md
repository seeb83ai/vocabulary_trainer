# Move vocabulary entry training initialisation into the store layer

Status: ready-for-agent
Category: enhancement

## What to build

When `StartTraining = true` on a word create request, the handler currently orchestrates three sequential calls after `CreateWord`: `AcknowledgeWord`, then a handler-side helper `initComponents` (defined in `handlers/words.go`), then async TTS. The invariant "a training-ready vocabulary entry is fully initialised" is owned by the handler, not the store — meaning any new initialisation step (e.g. mnemonic seeding) must be added to the handler.

Move the full initialisation into the store: `CreateWord` with `StartTraining = true` should atomically run `AcknowledgeWord` and `InitComponentsForWord` inside the same transaction. The handler calls `CreateWord`, gets an ID, fires TTS — nothing else.

The handler-side `initComponents` helper is deleted after the store takes ownership.

## Acceptance criteria

- [ ] `db.CreateWord` accepts a `startTraining bool` field (or the existing `CreateWordRequest.StartTraining` flag) and, when true, runs acknowledgement and component initialisation within the transaction
- [ ] The handler-side `initComponents` helper in `service/handlers/words.go` is deleted
- [ ] `AcknowledgeWord` call is removed from the handler's `Create` method (now happens inside `CreateWord`)
- [ ] Unit test in `service/db/db_test.go` asserts that after `CreateWord` with `StartTraining=true`, SM-2 progress exists and component progress rows are present for the vocabulary entry
- [ ] Unit test asserts that after `CreateWord` with `StartTraining=false`, no component progress rows are created
- [ ] All existing Go tests pass (`go test ./... -count=1`)

## Agent Brief

**Category:** enhancement
**Summary:** Move vocabulary entry training initialisation (SM-2 seeding, component initialisation) into the store's `CreateWord` transaction so the handler only fires TTS after creation.

**Current behavior:**
When a vocabulary entry is created with `StartTraining = true`, the handler performs three sequential calls after `CreateWord` returns: `AcknowledgeWord` (marks the entry as seen, setting `first_seen_date`), a handler-side helper `initComponents` (fetches SM-2 progress, calls `InitComponentsForWord`), and then async TTS audio generation. The invariant "a training-ready vocabulary entry has SM-2 progress, an acknowledgement date, and component progress rows" is enforced in the handler layer — not the store. Any future initialisation step (e.g. mnemonic seeding) must be added to the handler.

**Desired behavior:**
`Store.CreateWord` with `StartTraining = true` atomically runs all initialisation inside the DB transaction: acknowledgement (`AcknowledgeWord` logic), and component progress initialisation (`InitComponentsForWord`). After `CreateWord` returns, the handler calls `CreateWord`, receives a vocabulary entry ID, fires async TTS — nothing else. The handler-side `initComponents` helper is deleted.

**Key interfaces:**
- `Store.CreateWord(ctx, userID, req CreateWordRequest) (int64, error)` — `CreateWordRequest` already has a `StartTraining bool` field; when true, the method now atomically includes acknowledgement and component initialisation in the same transaction
- `Store.AcknowledgeWord(ctx, userID, wordID)` — its logic is inlined into the `CreateWord` transaction when `StartTraining=true`; the standalone method may remain for the separate "acknowledge from vocab list" flow
- `Store.InitComponentsForWord(ctx, userID, zhText, dueDate)` — called from within `CreateWord` when `StartTraining=true`; no signature change
- Handler `Create` method — remove the `AcknowledgeWord` call and the `initComponents` call; keep only `CreateWord` + async TTS

- [ ] `Store.CreateWord` with `StartTraining=true` atomically runs acknowledgement and component initialisation inside the transaction

- [ ] A DB-layer unit test confirms that after `CreateWord` with `StartTraining=false`, `first_seen_date` is null and no component progress rows exist
